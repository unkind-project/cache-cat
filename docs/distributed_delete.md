# 时钟设计草案

## 问题描述

现在要实现Raft版本的Redis。现在有俩个Redis命令。分别为

set key value px [固定的过期时间]

在固定的过期时间后销毁value。

set key value nx

如果没有key就设置value，有key就什么都不做。（类似的RMW的命令Redis有很多，这里只是举例）

***

在主从同步的情况下。当主节点在执行set px指令，并把指令推送到从节点时。从节点由于时钟和主节点存在偏差，实际上过期的时间和主节点存在区别。

假设这里主节点已经过期了，从节点还没有过期。那么此时如果set px指令到来，就会在俩个节点上产生不一样的效果。

这本质上是由于，任何时钟相关操作都可以看作是一次systemcall，而systemcall会带来不/稳定性，而Raft的状态机迭代是需要确定性进行的，相同的指令序列必须产生相同的结果。另一个角度看，分布式系统中是不存在绝对同步的时钟的。因为指令依赖了每个节点都不相同的时钟，导致俩个指令即便是以相同的顺序串行执行，也会让俩边的状态机最终不一致。然而Raft的状态机建模需要俩边状态机应用相同的指令，最终产生相同的确定性的结果。

## 基础假设和背景

> 时钟偏移（clock offset或Skew）和时钟漂移（clock drift）是不同的概念
>
> clock offset：某一时刻两个时钟之间的时间差。
>
> clock drift：时钟运行速率相对于理想时间的偏差，即“走得快还是慢”。

cache-cat整体指令的吞吐量超过30万每秒，使用raft协议。

此外，在没有故障的情况下，一条log通常能在毫秒级别完成提交。

对数据的定时过期逻辑，我们通常允许有几十毫秒的偏差。（毕竟机器自己运行时间长了也会产生时钟偏差）。

在读取中cache-cat已经使用了租约读来优化读取的性能。因此我们假设有界的时钟漂移。租约读在TiDB，ETCD等都非常常见。

此外设计需要保证对外语义是强一致的（和Raft设计一样，只暴露单个状态机的语义）。不能出现设置了key过期后，第一次读取key已经过期，第二次读取key又存在了等类似场景。

读从节点本身就只能实现最终一致，因此不需要考虑读从节点的强一致性。

## 参考方案

> 由于cache-cat是raft算法，以下方案设计均已经通过raft改造

### 确定性指令方面

该方案会对指令的原子性产生问题。因此该方案不做考虑。

先读取主节点当前状态机状态。生成确定性的指令。之后直接提交确定性的指令。比如把set nx直接变成一个确定性的set指令。

但是读取这一操作，之后状态机可能被其他日志修改（唯一能修改状态机的就是raft的日志apply），因此指令的原子性产生了问题。俩条set
nx指令同时进行，结果可能是都成功了。除非在读取之后，当前操作是第一条要被提交的操作。

### 删除时发起额外决议

> ETCD，Zookeeper，Consul等用的都是该设计思想实现定时删除。但是它们等并不是为了缓存设计，不需要频繁的删除设计。

每次定时删，都要发起一次Raft的共识，由主节点推送给从节点，只有共识完成，删除操作才能被完成。

当读取到一条已经被删除的数据的时候（对应Redis的惰性删），cache-cat需要先发起一次共识，删除当前数据，然后才能响应读请求，以避免产生一致性问题。这会导致读延迟上升。

>
这在Redis中并没有问题，这是因为Redis发给从节点的命令是不需要回复的，直接流式写入从节点即可，不需要等待从节点执行完毕再返回给客户端（无背压机制）。但是在Raft中AppendEntries执行完毕是需要大部分节点落盘并回复的，这会拖慢对客户端的响应。

缺点：大大影响了性能，设置过期时间在缓存系统中是高频操作。虽然这个操作是主节点自身发起，而不是通过客户端发起少了一次RTT，但是总的来说可能让raft需要提交的log数量接近翻倍，这会大大影响整体的吞吐量。

优化点：在进行定期删除的时候，可以将多条指令打包为一个操作，然后进行提交，来减少性能开销。

### Lamport时钟方案

既然不存在绝对的时钟，那么就将时钟本身也作为一个确定性的参数。主节点每10ms，**将自身当前的时间作为一个log提交作为一个逻辑时间，下文把该操作称之为时间决议
**。无论是定时删还是惰性删逻辑，都通过该时钟来确定。也就是所有的过期时间不参考本地不稳定的相对时钟，而参考被共识算法确定的逻辑时间。

消除了所有不确定因素，最终得到的结果一定是确定的。该方案还能有效避免时钟回拨。

缺陷：过期时间会有偏差。频繁的提交时间会导致机器空载的时候也要不停的工作。空载的时候也会出现每秒100次的日志提交到磁盘。

然而有趣的是当主节点一直有log被提交给从节点，那么主节点的心跳可以被省略。Raft默认情况下每50ms自动发送一次心跳给从节点。

## 共识时钟方案（Consensus Clock）

该方案为自研方案，最终被Cache-cat采用。

> 根据调研，无论是ETCD，Zookeeper，还是Redis等基建，都不会假设，主节点和从节点之间的时间偏移是有界的。
>
> 然而TIDB的PD集群，Cockroachdb会有类似的隐含假设（因为他们用了HLC方案）。

这里依然假设主节点和从节点之间的时间偏移（offset）是有界的。且在这个时间偏移范围内，不可能选出新的leader。比如这个时间小于1秒。

> 选主时间>时钟偏移时间

进行写入操作（raft需要提交日志的操作）的时候，主节点用本地的时间来给当前操作生成一个写逻辑时间戳时间戳取max(
写逻辑时间，读逻辑时间,物理时间)
。所有操作都会附带这个时间戳，从节点收到日志后需要用这个时间戳来更新自己本地的逻辑时间状态用来处理请求。即每一个操作开始前都要用日志中的逻辑时间来跟新自己本地的逻辑时间。因此，所有写操作都是确定性的。相当于推进逻辑时间这一个操作被附加到了每个entry中，这个操作实际上的开销是可以忽略不计的，它只是对本地一个int值做了修改。

**核心缓存的Map的过期时间，从原本的依赖物理时钟改为依赖写逻辑时钟。**

显然的，对于写操作，主节点和从节点都直接用逻辑时间戳来处理该请求。所有的惰性删和定时删都需要用状态机中的逻辑时间来确定性进行，写操作的一致性是容易被证明的：状态机的迭代是通过日志来确定性进行的。

读请求是该算法优化的重点，读同样需要生成一个读逻辑时间，这个时钟戳同样取max(写逻辑时间，读逻辑时间,物理时间)
。与写时间戳不同，读时间戳不需要经过决议才能更新，可以直接在内存中维护，每次读取就产生更新。显然这个读时间戳在单个节点上是只增不减的（不会产生回拨问题）。并且由于clock
offset是有界的，选主时间>时钟偏移时间，因此在新的主节点被选出时，新的主节点的本地时间一定大于之前的主节点的本地时间，因此全局上来看读请求生成的逻辑时间是只增不减的。
读逻辑时间不会让pair在内存中被删除，但如果 过期时间<读逻辑时间戳 该 pair则被视为不存在。

主节点处理读请求：如果主节点读到这条记录被删除了，那么即便主节点挂了。新的主被选出后，读从节点，这条记录仍然是被删除了（时间到了，时钟的偏移是有界的）。

然而从节点无法定时进行删除操作。当主节点发现有新的key删除了，并且已经超过10秒没发其他写操作的消息给从节点了。主节点就提交一次空日志推动写逻辑时间同步，该同步是为了GC清除过期的数据。这样就能避免没有外部请求的情况下，主从节点仍然要进行高频的日志提交，并且主节点和从节点的GC效率就只差10秒。此外没有key删除的情况下，就不需要将消息发给从节点。

> 如果主节点和从节点不走Raft来推进逻辑时间戳（比如用心跳），有可能出现从节点由于长时间无法和主节点取得联系，删除了过期条目。但是主节点生成了set
> nx的命令，这条命令在没有被过期的时候在主节点执行，但在从节点自己删除了之后才发给从节点，这时从节点和主节点执行的结果。可能产生二义性。

问题：此外该方案需要类似Google的TrueTime API来保证节点之间的绝对时间偏移在可接受范围内，不过这里可以接受几百毫秒的偏差，因此通常NTP授时就已经足够了。

## 算法证明

### **枚举所有的读情况**

> 二义性是主从切换时，俩边的本地时间无法达成共识导致的。但是主从切换后，新主节点的本地时钟一定大于旧主节点挂掉之前的本地时钟（选主时间>时间差），时钟是持续向前推进的。因此对外暴露的读操作语义在时间层面上一直是向前推进的。

读取主节点存在三种可能

1. 读到没数据的（无返回）：这种情况没有问题，因为数据要么就从来不存在，要么已经被其他写入操作导致的时间流动确定性的删除了（这个删除已经达成了共识）。
2. 读到有数据 但是过期的（无返回）
3. 读到有数据 且没有过期的（返回）

***

2 分为俩种情况：

- 主节点和从节点都删除了：这种情况没有问题是一致的
- 主节点已经删除，从节点还未删除，因为主节点和从节点的时钟偏移是有界的，如果此时主从不切换自然没有问题，但是如果主从切换了。由于切换选举的时间大于节点之间的时钟偏差。因此新的主节点被选出之后，读取操作返回的肯定也是过期的。

***

3 分为俩种情况

- 从节稍后先于主节点过期。如果主节点宕机新主节点被选出，读取新主节点发现已经过期了，但是这个case并没有影响。因为主节点本身也快要到期了。节点之间由于时钟偏移导致的过期时间存在差距本身就是可以接受的。
- 从节稍后于主节点过期。此时主节点宕机，新主节点被选出，新的主节点就会正常返回值。因此也没问题。

### 详细证明

证明目标：当一个操作判断一个key过期后，后续操作，都会将该key视为过期的。

当一个需要过期的pair被set的时候，它会存储一个绝对时间点，作为他的过期时间。

这里将需要修改状态机状态的操作统称为写操作。许多写操作是RMW（read-modify-write）的，也需要进行读取。例如Redis的append命令，需要先读取当前的pair之后才能对该pair的value进行修改。

对于写操作：过期时间<写逻辑时间戳 则被视为过期。这个写逻辑时间被通过raft提交当状态机中。生成时取max（写逻辑时间，读逻辑时间，物理时间）最大值。

对于读操作：过期时间<读逻辑时间戳 则被视为过期（不发起额外的共识）。读操作的逻辑时间被记录在内存中。生成时取max（写逻辑时间，读逻辑时间，物理时间）最大值。

写操作的逻辑时间显然是递增的，因为每次写操作产生的改变都被提交到了raft中，只有走完完整的决议才会生效。

在单机上读操作显然是递增的。当节点发生切换时，由于cache-cat要求主从切换的时间消耗大于所有节点之间的clock
offset（通过配置raft选举超时时间即可实现，通常不会超过1S）。因此读操作的逻辑时间即便在发生节点切换时也是递增的：选出下一个主节点的时候，新主节点的本机物理时间一定大于旧主节点的物理时间。

> 一些细节：
> 读逻辑时钟和写逻辑时钟是并发的，这意味着当读逻辑时钟生成时，另一个线程可能生成另一个更大的写逻辑时钟。但这其实不会有任何的影响，因为在并发的场景下，我们不要求读写的顺序性，即：无论哪个拿到的序列号更大都是可以接受的。

我们可以简单推出，

- 写请求之后收到的写请求，生产的逻辑时间一定大于之前的写请求。（这是通过raft状态机迭代保证的）
- 读请求返回之后收到的写请求，生成的逻辑时间一定大于写请求。
- 同理 写请求返回之后收到的读请求，生成的逻辑时间一定大于读请求。
- 同理 读请求返回之后收到的读请求，生成的逻辑时间一定大于之前的读请求。

显然

- 当一个写操作，返回一条数据已经过期时，接下来的写操作一定也认为该数据过期了。
- 当一个读操作，返回一条数据已经过期时，接下来的写操作一定认为该数据已经过期了
- 当一个写操作判断数据已经过期时，后续的读操作一定也认为该数据过期了。
- 当一个读操作返回数据过期时，接下来的读操作一定认为该数据已经过期了。
