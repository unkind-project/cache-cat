# Transaction设计

该方案旨在设计一套兼容Redis Transaction命令的框架。

## 事务

在开启事务后，服务端会将客户端发过来的一系列命令打包一个batch进行提交。因此中间不会插入其他命令。写命令的性能不会产生太大的影响，但是读命令由于需要被打包为一条条entry提交进行共识，因此性能可能会退化到和写命令类似。

## Watch

每个客户端保存 当前客户端watch了哪些key。类型为(key,db)这样元素的数组。

每个db都维护一个哈希表，记录key到客户端列表的映射。同一个key可能被多个客户端watch。

在所有的写命令路径里调用函数：找到哪些客户端watch了这个key，然后给这些客户端打上标记。

每个客户端都会有一个bool类型的flag，默认为true，如果被watch触发了就设置为false。

当exec提交时，无论是否提交成功，都会重新把这个flag设置为true。

当客户端关闭时，遍历当前客户端的所有key和对应的db，将db中的自己给删除。
