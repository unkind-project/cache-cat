# Transaction设计

该方案旨在设计一套兼容Redis Transaction命令的框架。

## 事务

在开启事务后，服务端会将客户端发过来的一系列命令打包一个batch进行提交。因此中间不会插入其他命令。写命令的性能不会产生太大的影响，但是读命令由于需要被打包为一条条entry提交进行共识，因此性能可能会退化到和写命令类似。

## Watch

watch命令的所有状态是挂在主节点上的，并不产生任何的复制。但是exec命令判断是否执行，确要根据watch的结果。因此watch命令的适配是困难的。换句话说从节点可能不知道主节点的watch情况。

如果exec在提交到状态机之前就完成了计算。在状态机队列中其他被提交的任务可能修改watch的key。
