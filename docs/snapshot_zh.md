# 快照逻辑

## 常见的快照策略

Consul，使用go-memdb进行存储，其实现了内存mvcc机制。Zookeeper，将所有操作变成CAS操作，因此后续的请求可以被应用多次，从而达到相同的数据逻辑。

Dragonflydb，通过任期手动标识哪些数据是旧的哪些数据是新的，执行完快照逻辑后手动收集变动部分数据。ETCD，通过存储层的数据库（BoltDB）来直接实现MVCC快照。Redis
通过linux下的fork指令（linux的fork会复制整个进程的地址空间，但只复制调用fork的那个线程）。

然而以上数据结构都存在特定的问题。Consul，Zookeeper，ETCD 只支持非常单一且特定的数据结构。
Kvrocks，pikiwidb，Tendis均采用Rocksdb，直接使用Rocksdb的MVCC机制来实现快照隔离。
而Redis的fork存在问题，当快照期间遇到大规模的数据写入时，fork操作会产生大量的内存占用。这是因为fork虽然是COW的，但是每一个值的修改，都会导致完整的内存页进行一次复制（在x64系统下通常是4kb）。并且只要父进程持续写入，就会不停的发生页复制，如果磁盘性能不够强会导致snapshot花费很久时间加剧了页复制的情况。
此外fork操作还会造成一个短暂的STW停顿，主进程在内核中执行fork的一段时间无法处理对外的客户端请求。生产上为了减小fork的影响，会限制Redis的容量上限。

Dragonflydb避免了fork操作。https://www.dragonflydb.io/docs/managing-dragonfly/snapshotting
。但引入了新的问题，https://github.com/dragonflydb/dragonfly/issues/6578 。
Dragonflydb为hashmap定制了一个通用的快照策略，因此能兼容Redis的所有数据结构。然而，当map中的一个key的value是一个较大的值时，由于读写操作不能同时访问该key，从而导致对该键值对进行快照会消耗较长的时间。
Dragonflydb内部使用了一个fiber，这个fiber只能自己放弃cpu或者任务出现阻塞才会产生切换。（fork和主行线程其实都是一个线程）
试想你将Dragonflydb的List结构用作消息队列。当某个key的value有几十G时，此时进行快照会长时间对该键值对进行占用，写入操作只能在原地等待直到对于该键值对的快照完成。这是由于Dragonflydb目前没有为所有的数据结构提供专属的快照策略导致的。
仅针对于Hashmap的读写并发结构的设计是相对简单的。但当Hashmap的value为各种不同的如链表，Skiplist等数据结构时，设计的复杂度会变得非常高。

此外Dragonfly的策略会让快照期间正在写入的线程去序列化数据的（这增加了延迟）。并且每次这个序列化操作都是直接序列化key下的整个value。因此Dragonflydb中不能在value中存放大尺寸的数据。

在没有MVCC，不采用Fork指令的情况下，只有Dragonflydb的策略和Zookeeper的策略可以被参考。下文对比俩方的缺点。默认Zookeeper是让每个指令进行CAS操作，我们这里假设只在快照的期间记录版本号。

Dragonflydb的缺点

- 当快照线程还没有来得及序列化数据的时候，每次产生的写操作都会导致先进行阻塞，完成序列化发送给落盘线程。带来了潜在的延迟变高。
- 由于默认是将数据统一先序列化，然后发送快照线程，会带来额外的内存占用（并且在理论上就无法存放过大的value值，比如当单个List中有数十G的value）。即便这些内存在写入完毕后就会被释放。Zookeeper可以一边遍历一边写入磁盘。

Zookeeper方案缺点(下文方案)

- 所有版本数据需要在快照中存在。（内存部分俩者是一致的，都需要维护一个u32的版本号，但是Dragonfly的方案理论上不需要将这个版本号写入磁盘）
- 在快照安装时，需要对每一条写入操作的指令实现对应的CAS操作，这会大大增加适配的复杂度。
- 无法和兼容Redis的快照格式。
- 快照构建时，所有数据都需要被缓存起来。此时所有的请求都会带来翻倍内存占用（Zookeeper自身的共识算法能避免这个问题）。

## 快照策略

> 下文是Cache-cat原创的快照策略，总思想类似Zookeeper，将所有操作变成一个CAS操作。对应Dragonfly中的Relaxed快照策略。Zookeeper方案有个缺点，RAFT的快照中的快照语义需要是一个时刻的瞬时快照。

其中map中的每个value都可以被试做为一个原子访问单元。对这个value的任何操作都是原子的，且读写并发的。

我们依然要记录last_applied_log_id（快照产生时最后应用的日志id），last_membership（最后的成员配置，raft自用不用管），snapshot_num（快照编号
u32），snapshot_state（当前快照状态）。这些数据都被一个锁保护统称为meta_data，并且被一个parking_lot::mutex（待讨论）保护起来。

此外每一个value都会在内部维护一个版本号 u32类型。
**业务线程**

1. 当有一批新的操作到来。通过锁获取meta_data。
2. 更新last_applied_log_id
3. 如果当前snapshot状态为false。直接将数据写入cache_map结束。
4. 如果当前snapshot状态为true：获取原来数据的版本号，如果没有旧的版本号，则版本被视为0。将版本号加1，写入cache_map中。最后将老版本号和当前操作写入队列中暂存。（将所有写入操作变为一个CAS操作）

**快照线程**

获取meta_data的锁，将快照标记为开始，并且保存meta_data元数据。释放锁。

进行快照操作迭代所有数据，将数据写入磁盘中。

获取meta_data锁，将快照标记为结束。理论上这里获取过锁，因此不会出现close-and-drain race问题。读取删除队列中的全部数据，写入磁盘。

**恢复操作**
读取全量数据，恢复到状态机。（这时暂停对外访问）
读取队列中的增量数据，对每一条增量数据进行CAS操作。如果当前数据的版本号和队列中的版本号一样。

为什么这样是对的？快照线程和业务线程是并发的。因此快照线程在执行快照的时候会保存俩种快照数据，旧的和新的（每条数据都是原子的）。
当快照开始的时候业务线程执行的所有操作会被存下来到队列中，保存下来的所有队列元素都是带版本号的CAS操作。
因此在恢复快照的时候，只需要将快照数据恢复+队列中的数据进行CAS，最终得到的就是正确的最新数据。
快照线程在开始和结束的时候都会短暂的对状态机进行加锁，最终完整被还原后的数据是第二次加锁时内存中的数据。因此这种快照策略通常被视为后快照策略。
存在的问题：

需要反序列化每条操作，对每条操作进行定制的指令。

## 读写逻辑时钟兼容

> 问题：当一个操作被被push到队列时，如何知道pair+当前操作后的结果？ 如果主节点执行该操作，可能在key过期之前来执行，也可能在key没有过期的时候来执行。

当快照开始后，每一条操作的进行都会被push到操作队列中，并且将生成的写逻辑时钟，push到队列中（这个写逻辑时钟是已经计算完`max(写逻辑时钟，读逻辑时钟，当前时间戳)`之后确定性的结果）。

**恢复快照**

最开始是apply全量的状态机，状态机初始化的时候这个写逻辑时间戳是0 。因此没有数据会被过期。接下来是apply增量的队列。

之后还原的时候先还原写队列中的逻辑时钟，然后直接apply这个操作序列。写逻辑时钟是操作队列，并且每个操作都是确定性的。当apply第一条队列中的日志时，写逻辑时间戳将会被更新。这时数据开始走正常的过期逻辑。每个操作执行完毕后，哪些数据应该被过期都是确定性的。

日志apply的顺序依旧和主节点是一样的，只不过如果发现版本不对就不会实际修改数据。

我们记录的操作队列是有序的，因此在恢复快照的时候，只需要将快照数据恢复+队列中的数据进行CAS，最终得到的就是正确最新的数据。

最后快照完毕时，也会将当前节点的元数据以及写逻辑时间戳加锁后保存。
